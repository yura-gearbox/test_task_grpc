## 2. СТАТУС ПОКРЫТИЯ ПО БЛОКАМ

### Блок 1: Lifecycle & Smoke (Жизненный цикл процесса)

* **TC-LC-01 (Liveness Probe):** **PASSED**. gRPC метод `IsReady()` стабильно возвращает готовность после инициализации.
* **TC-LC-02 (SIGINT Shutdown):** **FAILED**. Процесс корректно освобождает ресурсы, но падает с системным кодом `-2` вместо задекларированного `0`.
* **TC-LC-03 (SIGTERM Shutdown):** **FAILED**. Процесс корректно освобождает ресурсы, но падает с системным кодом `-15` вместо задекларированного `0`.

### Блок 2: Specification Validation (Валидация по спецификации)

* **Packet Counter (`TC-SV-PK-01` -> `TC-SV-PK-03`):** **PASSED**. На диапазоне `[0...100]` пакетов инкремент и накопление работают со 100% точностью.
* **Byte Counter `'A'` (`TC-SV-BY-01` -> `TC-SV-BY-05`):** **PASSED**. Фильтрация шумовых ASCII-символов работает корректно. Побайтовый разбор буфера внутри одной датаграммы и накопление между разными пакетами выполняются без искажений.

### Блок 3: Implementation Verification (Тестирование белого ящика)

* **TC-IV-PK-01 (Пакеты = 255):** **PASSED**. Верхняя граница эквивалентного класса пройдена успешно.
* **TC-IV-PK-02 (Пакеты = 256):** **FAILED**. Зафиксирован сброс счетчика пакетов в `0` из-за переполнения `uint8_t` (ожидалось 256).
* **TC-IV-BY-01 (Байты = 255):** **PASSED**. Верхняя граница побайтового парсинга пройдена успешно.
* **TC-IV-BY-02 (Байты = 256):** **FAILED**. Зафиксирован сброс счетчика байт в `0` из-за переполнения `uint8_t` (ожидалось 256).

### Блок 4: Non-Functional / Stress (Производительность)

* **TC-NF-01 (Burst Rate Нагрузка):** **PASSED**. При импульсной отправке пачки из 255 пакетов на максимальной скорости процессора **Packet Loss Rate = 0%**. UDP-слушатель успешно справляется с плотным потоком данных на loopback-интерфейсе, данные в буфере ОС не теряются.

## 3. РЕЕСТР НАЙДЕННЫХ ДЕФЕКТОВ (BUG LOG)

### BUG-01 [Severity: CRITICAL]: Аппаратное переполнение (Overflow) счетчиков статистики

* **Симптомы:** При отправке 256-го UDP-пакета или получении 256-го целевого символа `'A'` счетчики в gRPC-ответе сбрасываются в `0`.
* **Причина (Белый ящик):** В файле `main.cpp` переменные `packet_count` и `a_bytes_count` объявлены как `std::atomic<uint8_t>`. Тип данных `uint8_t` имеет максимальную емкость 255. При попытке записать значение 256 происходит циклический сдвиг битов (wrap-around) на уровне процессора. Это нарушает контракт `.proto`-файла, где данные задекларированы как `uint64`.
* **Рекомендация:** Заменить тип атомиков в C++ коде на `std::atomic<uint64_t>`.

### BUG-02 [Severity: MINOR]: Некорректный код возврата процесса при штатном завершении приложения

* **Симптомы:** При штатном останове приложения системными сигналами `SIGINT` и `SIGTERM` процесс возвращает коды `-2` и `-15` соответственно, вместо кода `0`.

* **Причина (Белый ящик):** Предположительно, причина в том, что перехват завершающих сигналов происходит в фоновом потоке (`udp_thread`). Когда фоновый поток закрывает сокеты и завершается, главный поток всё ещё ждет закрытия фонового потока (`udp_thread.join()`). ОС считает это зависанием и принудительно завершает процесс с ненулевым кодом ошибки.

* **Рекомендация:** Перенести перехват и обработку сигналов операционной системы в главный поток.

## 4. ЗАКЛЮЧЕНИЕ И РЕКОМЕНДАЦИИ (CONCLUSION)

Архитектура UDP-слушателя написана качественно и демонстрирует отличную пропускную способность под импульсной нагрузкой без потери датаграмм (Packet Loss = 0%). gRPC-интерфейс работает стабильно.

Однако приложение **не может быть допущено к эксплуатации** из-за критического бага переполнения счетчиков (BUG-01). После исправления типов данных на `uint64_t` и переноса обработки сигналов в главный поток (BUG-02), требуется повторный полный запуск разработанного регрессионного сьюта автотестов. Все 16 сценариев должны завершиться со статусом PASSED, подтверждая исправление дефектов.
