# Документация тестирования: Методология, Стратегия и Матрица сценариев

Данный документ определяет методологический подход, архитектурную стратегию и детальный план проверок для системного тестирования C++ приложения `grpc_udp_monitor`.

---

## ЧАСТЬ 1. МЕТОДОЛОГИЯ ТЕСТИРОВАНИЯ (TEST METHODOLOGY)

### 1. Уровни тестирования

- **Системное тестирование (System Testing):** Выбран как единственный применимый уровень. Система тестируется как единое целое в изолированном контуре.
- *Исключенные уровни:* Модульное (Unit) тестирование исключено (зона ответственности разработки); Интеграционное тестирование неприменимо (компоненты gRPC и UDP не разделены на независимые сервисы); E2E-тестирование невозможно из-за отсутствия сквозных бизнес-сценариев верхнего уровня.

### 2. Техники тест-дизайна

В рамках проекта применяется комбинированный подход, сочетающий методы серого и белого ящика:

* **Тестирование по спецификации (Серый ящик):** Анализ `.proto`-контракта. Для полей типа `uint64` определены базовые эквивалентные классы и граничные значения: `0` (начальное состояние), `1` (минимальный шаг инкремента) и `100` (стабильное накопление данных в рамках валидного диапазона).
* **Тестирование по реализации (Белый ящик):** Анализ исходного кода `main.cpp` выявил архитектурный дефект - внутренние счетчики имеют тип `uint8_t`, несмотря на декларацию `uint64` в Protobuf. Такой тип приведет к переполнению при значениях свыше `255`. Спроектирован класс проверок аппаратного переполнения (overflow) на значении `256`.
* **Стресс-тест UDP-листенера:** отправка `255` пакетов без пауз, с упором на высокую интенсивность. Цель - проверить, что слушатель справляется с плотным потоком данных без потерь данных.

### 3. Критерии успешности

1. **Функциональное системное тестирование:** Покрытие 100% заложенных сценариев Happy Path и граничных значений. Все тесты завершены со статусом `PASSED`. Точность подсчета со стороны gRPC составляет 100% относительно отправленного объема. Приложение штатно завершает работу по системным сигналам с кодом `0`.
2. **Стресс-тестирование (Производительность):** При импульсной нагрузке показатель потери пакетов должен быть строго равен `0%`. Любая потеря данных интерпретируется как дефект производительности приложения и является блокирующим фактором для перехода на этап сетевого тестирования..
3. **Тестирование белого ящика:** Проверки гарантируют локализацию дефектов типов данных. Все внутренние типы данных счетчиков логики приложения должны гарантированно обеспечивать стабильную работу и корректное накопление статистики на всем заявленном в .proto-контракте диапазоне значений, без рисков внутренних переполнений или искажения данных.

## ЧАСТЬ 2. СТРАТЕГИЯ ТЕСТИРОВАНИЯ

### 1. Область тестирования

* **В рамках тестирования:**
  * Валидация жизненного цикла процесса (старт, проверка готовности gRPC-сервера, штатный останов).
  * Валидация gRPC-методов `IsReady` и `GetUdpStatistics`.
  * Проверка точности подсчета входящего UDP-потока датаграмм на порту `2032`.
  * Верификация побайтовой фильтрации целевых символов `'A'` внутри пакетов.
* **Вне рамок тестирования:** На этапе системного тестирования надежность и производительность компиляторов и внешних библиотек (`Boost.Asio`, `grpcio`, `protoc`) принимается как доверенная.

### 2. Тестовое окружение

* **Инфраструктура:** Изолированный Docker-контейнер на базе Ubuntu 24.04 LTS. Цикл выполнения полностью изолирован, а запуск тестов производится вручную, поэтому для их старта и просмотра результатов требуется подключение к терминалу контейнера.
* **Сетевой уровень:** Взаимодействие тестового каркаса (Python) и заглушек с целевым бинарником ограничено внутренним loopback-интерфейсом (`127.0.0.1`). Проброс портов наружу отключен.

### 3. Стек инструментов

* **Python 3.12.3:** Базовый язык. Версия соответствует требованиям ТЗ и является стандартной для Ubuntu 24.04.
* **Pytest 9.0.3:** Тестовый фреймворк. Выбран согласно требованиям ТЗ.
* **grpcio / grpcio-tools:** Официальный стек библиотек для генерации стабов и gRPC-взаимодействия.
* **socket & subprocess:** Встроенные модули Python для отправки UDP-датаграмм и контроля жизненного цикла ОС-процессов.

### 4. Стратегия тестовых данных

* **Структура датаграммы:** Сырой массив байт переменного размера (от `1` до `256` байт, в зависимости от сценария). Состоит из целевых данных (символы `'A'`, ASCII `0x41`) и шумовых данных (любые ASCII символы, кроме `'A'`).
* **Режимы нагрузки:**
  * *Режим верификации логики:* Пачки пакетов объемом до `256` штук. Искусственная межпакетная задержка `2 мс` для гарантированной обработки операционной системой.
  * *Режим импульсной нагрузки (Burst):* Пачка из 255 пакетов, отправляемая в циклическом потоке на максимальной скорости ядра процессора без пауз.
